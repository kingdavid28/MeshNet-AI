1 · Security
Critical IAM API key exposed in the browser
useCloudantNodes.ts reads VITE_CLOUDANT_API_KEY and sends it directly in the Authorization: Bearer header from the client. Vite bakes every VITE_* variable into the JS bundle at build time — anyone who opens DevTools can read the key, rotate it, or call Cloudant directly.
Fix: Move all Cloudant calls to the Express backend. The frontend never touches Cloudant credentials directly.
Critical CORS set to origin: "*" on all layers
Both the Express backend (backend/src/index.ts) and the FastAPI server (api_server.py) allow any origin. In a disaster-response deployment the backend is typically on the LAN and only the app's WebView or a known dashboard host should be allowed.
Fix: Enumerate allowed origins via env var (CORS_ORIGINS). Wildcard may be acceptable for the Python service if it is never reachable from the public internet, but not for Express.
Critical Shared secret hard-coded as "CHANGE-ME-before-deployment"
config.py line 102: NODE_SHARED_SECRET = os.getenv("MESH_NODE_SECRET", "CHANGE-ME-before-deployment"). If the env var is not set in production the HMAC secret is publicly known from the source code.
Fix: Remove the fallback string and make the app refuse to start if MESH_NODE_SECRET is absent. Add it to a CI secret and a deployment checklist.
Critical Messages stored as plaintext in SQLite
The messages table column is named ciphertext but the CommsTab in App.tsx sends raw user text and the DB layer stores it verbatim. The RoutingEngine AES-256-GCM encryption exists in routing_engine.py but is never called by the frontend send path.
Fix: Implement client-side encryption before POSTing. Alternatively, have the Express backend encrypt on receipt — but then it holds the key, which is a different threat model. Decide and implement one of the two.
High No authentication on any Express endpoint
Any device on the same network can register arbitrary nodes, post fake alerts, or drain the SQLite database. The rate limiter (middleware/rateLimit.ts) is the only protection — it limits volume, not identity.
Fix: Add a shared API token (Bearer or HMAC-signed request) for node-to-backend calls. A per-device identity derived from the Capacitor plugin's stable node ID is a natural fit.
High BLE GATT characteristics have no authentication
MeshDiscoveryPlugin.kt exposes node ID, label, lat/lng, battery, and signal over open GATT characteristics (PERMISSION_READ only). Any BLE-capable device within range can connect and read all mesh node data without credentials.
Fix: Add a challenge-response over GATT before serving characteristics, or use GATT encryption (PERMISSION_READ_ENCRYPTED) which requires an OS-level BLE pairing.
Medium AES session key is never returned to the caller in the AI-route endpoint
api_server.py deliberately omits session_key from the /api/simulation/ai-route response — but provides no out-of-band mechanism for the rescue base camp to receive it. The encrypted hop packets are undecrytable without it.
Fix: Define and implement the key-distribution channel: pre-shared key per mission, QR code pairing at mission start, or asymmetric key wrap.
2 · Android Native Layer
Critical App has never been built or deployed to a physical device
android/app/src/main/assets/public/ was empty until cap sync was just run. There is no evidence of a successful Gradle build, APK signing, or a device test. BLE advertising, GATT server startup, and Wi-Fi Direct group creation have never been exercised on real hardware.
Fix: Run pnpm run cap:open, build a debug APK in Android Studio, sideload to a physical device (API 26+), and walk through the BLE + Wi-Fi Direct permission grant flow manually.
Critical Backend URL is hard-coded to emulator loopback (10.0.2.2)
MeshDiscoveryPlugin.kt line 125: private var apiBase = "http://10.0.2.2:4000". On a real device 10.0.2.2 does not resolve. The Capacitor config supplies the URL via startDiscovery(), but the heartbeat fires before the first call reaches the plugin only if the default is wrong.
Fix: Change the default to the machine's LAN IP (or make it blank and guard in patchHeartbeat() until overridden). Document in README with a note about VITE_API_BASE_URL.
High No Android launcher icons or splash screen
AndroidManifest.xml references @mipmap/ic_launcher and @style/AppTheme but neither the res/mipmap-*/ directories nor res/values/styles.xml exist in the project. The APK will fail to compile in Gradle.
Fix: Add the standard Capacitor Android resource scaffold: run npx @capacitor/assets generate after placing a 1024×1024 icon, or copy the Capacitor android-template res/ tree.
High GPS coordinates sent to BLE peers are 0.0, 0.0 until updated
useMeshDiscovery.ts initialises latRef/lngRef from deviceLocation which is null on first render. The initial registerSelf() and the GATT server characteristics (CHAR_LAT/CHAR_LNG) broadcast 0.0, 0.0 until GPS acquires — placing all newly-discovered nodes at the ocean off the coast of Africa on the map.
Fix: Gate registerSelf() behind a non-null location check, or display a clear "Acquiring GPS…" state and avoid registering the node until a first fix arrives.
Medium BLE GATT read is not thread-safe on older Android versions
processBleDiscovery() is called from within the GATT callback on the binder thread. The chars map is mutated in onCharacteristicRead() which runs on the same binder thread — but on Android 8–11 concurrent GATT callbacks for multiple devices can overlap, causing ConcurrentModificationException on the shared map.
Fix: Use a ConcurrentHashMap per-device, keyed by device address, or serialize all GATT reads through a single-threaded coroutine dispatcher.
Medium Wi-Fi Direct automatically connects to any device with "meshnet" in its name
onWifiPeersChanged() checks device.deviceName.contains("meshnet", ignoreCase = true). This is trivially spoofable — any device naming itself "meshnet-evil" would be invited to join the mesh group, gaining access to the local Express API at port 4000.
Fix: Replace the name-match with a GATT-verified node ID whitelist built during the BLE discovery phase. Only connect Wi-Fi Direct to addresses whose BLE identity has been verified.
3 · Networking & Offline Behaviour
High Alert send silently succeeds even when backend is unreachable
AlertTab in App.tsx: the catch block on the fetch call is empty — the UI shows "Alert Sent · Broadcast to 6 nodes" regardless of whether the backend received the alert. In a real disaster this creates false confidence.
Fix: Queue undelivered alerts in IndexedDB / localStorage and retry with a background sync event (PWA Background Sync API). Show a distinct "Queued — will send when connected" state.
High Express backend assumed reachable at a single static URL
Every data path (heartbeat, register, routing query, SSE stream) targets VITE_API_BASE_URL which is baked at build time. In a real mesh, the backend may be running on a different device at a different IP depending on which node is acting as group owner. The Wi-Fi Direct onWifiGroupConnected() overrides apiBase for that call only; the JS side never learns the dynamic IP.
Fix: Emit a backendUrl field in the wifiGroupFormed Capacitor event and update the apiBase state in useMeshDiscovery so all subsequent calls from the frontend target the correct peer.
Medium SSE stream has no auth and no node filter
useSignalStream.ts opens GET /api/signal/stream with no credentials. Anyone on the LAN can subscribe and see all flicker events for all nodes, including location data embedded in the topology.
Fix: Require the same API token as all other endpoints. Optionally add a ?nodeId= filter so each device only receives events relevant to itself.
Medium Comms messages are append-only with no delivery confirmation
Messages POSTed to /api/messages are written to SQLite and never forwarded over BLE or Wi-Fi Direct to the intended recipient. The "Encrypted · offline · peer-to-peer" label on the UI is aspirational — there is no actual P2P delivery path.
Fix: Implement hop-forwarding: when a message arrives at a node that is not the destination, the backend should PUT it on the outbound queue for BLE/Wi-Fi forwarding to a neighbour that is closer to the target.
4 · Backend & Data Layer
High SQLite file used as the single source of truth with no replication
All node state, edges, alerts, and messages live in a single meshnet.db file on whichever device runs the Express backend. If that device loses power, all shared state is gone. The WAL journal helps writes but does not provide HA or multi-node consensus.
Fix: For a field deployment, replicate to IBM Cloudant on every heartbeat (it already has write support — only reads are wired). Alternatively mirror the DB to a second relay node's Express instance.
High Node stale-detection and eviction are missing
Nodes registered via /api/mesh/register are never removed. A node that goes silent (battery died, left the area) stays in the topology indefinitely, polluting the routing graph and cluttering the map. The last_seen column is updated but never queried for eviction.
Fix: Add a cleanup job (30 s interval) that marks nodes as offline if last_seen is older than 3× the heartbeat interval, and removes them from the graph after 5 minutes.
Medium Two separate routing engines with diverging behaviour
route_engine.py (Dijkstra on live topology from Express) and routing_engine.py (battery-prioritised Dijkstra on the offline SQLite mesh) coexist but are not reconciled. The Express /api/route endpoint proxies only route_engine.py. routing_engine.py (with AES encryption) is only reachable via the Python /api/simulation/ai-route endpoint, which the frontend never calls.
Fix: Decide on one routing engine for production. Merge the battery-penalty logic from routing_engine.py into route_engine.py and expose encryption through the same path.
Medium GPS coordinates fixed to San Francisco in the AlertTab UI
Line 656 in App.tsx: "37.7749° N · 122.4194° W" is hard-coded text in the location badge. The actual useDeviceLocation hook is wired to the map but not to the alert form — every alert sent shows San Francisco coordinates regardless of where the device actually is.
Fix: Pass useDeviceLocation() into AlertTab and render the live coordinates (or "Acquiring…" if not yet resolved).
Low Node ID generated with Math.random()
useMeshDiscovery.ts: "node-" + Math.random().toString(16).slice(2, 8) — 6 hex chars = 24 bits of entropy. With 100 nodes the collision probability is about 0.05%, but uniqueness is not guaranteed across reinstalls.
Fix: Use crypto.randomUUID() (already available everywhere the app runs) and store the full UUID in localStorage.
5 · UX & Product Completeness
Medium HomeTab and MeshCanvas use hardcoded static data only
The NODES, EDGES, and MESSAGES constants at the top of App.tsx are rendered in the Home tab canvas (MeshCanvas) and message list. They do not reflect live backend data. The MapTab correctly uses useCloudantNodes but the Home tab does not.
Fix: Wire MeshCanvas to the same useCloudantNodes data source, or extract a shared context provider so all tabs see the same live node list.
Medium "Broadcast to 6 nodes" is always hardcoded
The post-send confirmation screen in AlertTab says "Broadcast to 6 nodes" unconditionally. In a real deployment this number should reflect the current live peer count from useCloudantNodes.
Fix: Read nodes.length from the shared node hook and interpolate it into the confirmation message.
Low No iOS support
MeshDiscoveryPlugin.ts documents "iOS: returns safe no-op stubs". There is no ios/ Capacitor project and no Swift plugin. Disaster teams often use iPhones — Core Bluetooth and MultipeerConnectivity are the iOS equivalents.
Fix: Run npx cap add ios and implement MeshDiscoveryPlugin.swift using CBCentralManager / CBPeripheralManager for BLE and MCSession for peer connectivity.
Low No accessibility (a11y) support
Canvas-rendered nodes (NodeMapCanvas.tsx, MeshCanvas), icon-only nav bar buttons, and colour-only status dots have no ARIA labels or roles. The app would fail WCAG 2.1 AA — a concern for government-funded disaster-response tools.
Fix: Add aria-label to all interactive elements; provide a text-based node list as a screen-reader alternative to the canvas map.
6 · Operations & Deployment
High No process management for the three server processes
Running MeshNet AI in production requires three concurrent processes: Vite (or the built PWA served statically), the Express Node.js backend, and the FastAPI Python server. There is no Procfile, docker-compose.yml, or systemd unit file. There is also no documented startup order (Express must start before FastAPI or the Python health check fails).
Fix: Add a docker-compose.yml with three services and health-check dependencies, or at minimum a Procfile for use with pm2 / foreman.
Medium No automated tests
There are no test files for the React frontend, the Express routes, or the Python routing engine (a backend/tests/ directory exists but is empty). The RouteEngine, SignalMonitor, and BurstDispatcher are all pure-logic classes that are straightforward to unit test.
Fix: Add pytest tests for Python routing logic and Vitest/Jest tests for the React hooks. Prioritise SignalMonitor.ingest_sample() and RouteEngine.query() as they contain the core decision logic.
Medium No APK signing configuration
android/app/build.gradle has no signingConfigs block. The Play Store requires a signed release APK; sideloading to a device requires at minimum a debug keystore. There is no CI workflow to build and sign the APK.
Fix: Add a signingConfigs.release block in build.gradle that reads keystore path/password from env vars, and add a GitHub Actions workflow that builds and signs the release APK.
Low Database migrations are ad-hoc
Schema changes are handled by try/catch ALTER TABLE statements at boot time in db.ts. This works for additive changes but cannot handle column renames, type changes, or data migrations.
Fix: Adopt a lightweight migration framework (e.g. better-sqlite3-migrations or numbered SQL files applied sequentially) so the schema history is reproducible.
Low Simulation seeding targets Manila; app UI targets Cebu
mesh_simulator.py and api_server.py /api/simulation/seed generate random nodes centred at Manila (14.5995°N, 120.9842°E). The seed data in useCloudantNodes.ts and the map centre target Cebu City (10.3157°N, 123.8854°E). Running a seed will populate the map with nodes ~600 km off-screen.
Fix: Align the simulation base coordinates to Cebu City, or make them configurable via env var.
