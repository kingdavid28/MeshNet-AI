01 What MeshNet AI Does

MeshNet AI is an emergency communication system that works when cell towers and internet are destroyed. Phones and laptops communicate directly with each other using Bluetooth and Wi-Fi — no tower, no router, no internet required.

🆘 For Victims
Send SOS alerts with GPS
Request medical help
Share your location
Signal "All Clear" when safe
Relay messages for others
📡 For Rescue Teams
See all nodes on live map
Find AI-optimised rescue routes
Trigger HQ emergency broadcast
Monitor battery & signal levels
Log all events in activity feed
Key fact: Every phone that joins the mesh automatically extends its range. 10 phones = 10× the coverage area. Phones near the edge relay messages for phones that cannot reach the relay directly.
02 Starting the System — Rescue Team
Before you start: You need Node.js and Python installed on the relay laptop. Do this setup before the disaster — not during.

Step 1 — Start all three servers

Open 3 separate terminal windows on the relay laptop and run one command in each:

# Terminal 1 — Python AI Routing Engine (port 5050)
cd "C:\...\Mobile App UI Design\backend"
python -m uvicorn api_server:app --port 5050 --reload

# Terminal 2 — Node.js REST API + SQLite (port 4000)
cd "C:\...\Mobile App UI Design\backend"
npm run dev

# Terminal 3 — Vite frontend (port 5173 dev / 4173 preview)
cd "C:\...\Mobile App UI Design"
pnpm dev
2
Open the dashboard in your browser
Navigate to http://localhost:5173 on the relay laptop. On a screen wider than 768px the Command Dashboard opens automatically. On a phone-sized screen, the Mobile App opens instead.
3
Confirm nodes appear on the map
The map should auto-fit to Cebu City and show 5 seed nodes as coloured dots. If the map is grey tiles, click the Fit Nodes button (grid icon, top-right of map). Wait up to 10 seconds for the first data poll.
4
Turn on the Wi-Fi hotspot for victim onboarding
On the relay phone/laptop, enable a Wi-Fi hotspot named MESHNET-xxxxxx (replace with a unique 6-char suffix). Victims nearby can connect and open the app in their browser at http://192.168.49.1:4173.
Tip: Print the IP address and hotspot name on a card or write it on a piece of paper. Stick it to the outside of the relay device so victims can read it without asking.
03 Mobile App — Victim / Field User

The mobile app has 4 tabs at the bottom. Here is what each one does:

🏠 Home
🔔 Alert
🗺 Map
💬 Comms

Home Tab

!
Big Orange SOS Button
Tap once to start a 3-second countdown. The button changes to "SENDING IN 3… 2… 1…" giving you time to cancel. After countdown the button goes red and broadcasts "SOS BROADCASTING" to all reachable nodes on the mesh. Your GPS is attached automatically.
Use this for: Any life-threatening emergency. Fastest way to call for help.
❤
Medical Request button
Flags a medical emergency at your location. Use this when someone is injured. The rescue team's dashboard shows medical alerts in orange.
📍
Share GPS button
Broadcasts your current coordinates to all nodes. Use this when you are moving to a new location and want rescue teams to track you.
✓
All Clear button
Signals that you are safe. Removes your SOS flag on the rescue map. Always send this once you are evacuated so rescue resources can be redirected.
⚡
Relay Mode button
Activates your phone as a relay node — boosting network range. Your device will forward messages from others. Costs more battery. Use when you are stationary with good signal.

Alert Tab

1
Select alert type
Choose from: War Zone, Flood, Medical, Fire, SOS, Evacuate. Tap the correct card — it highlights with the type's colour when selected.
2
Type your message (optional but recommended)
Describe the situation: location, number of people, severity. Max 280 characters. Example: "3 injured civilians at the school gymnasium — 1 critical, need ambulance"
3
Tap SEND SOS
The button glows red and activates only after you pick a type. Your GPS is automatically attached. The screen shows "SOS Sent — Broadcasting to all reachable mesh nodes" with a pulsing radio icon. The alert appears in the rescue team's Activity Log within seconds.

Map Tab

📍
View your position and all nearby nodes
The map shows all devices currently on the mesh as coloured dots on a real OpenStreetMap. Your location is shown as a blue pulsing circle (the "You" marker). Tap any dot to see that device's name, battery, signal, and radio protocol status.
⊕
Locate Me button (blue crosshair icon)
Appears only when GPS is active (blue "GPS ±Xm" badge visible). Tap to fly the map to your current position at zoom level 16.
⊞
Fit Nodes button (grid icon)
Zooms out to show all nodes on the map at once. Use this when you have navigated away and need to see the full mesh.

Comms Tab

✉
Send a text message to the mesh
Type any message in the input box at the bottom and press the orange send button (or press Enter). Your message is broadcast to all nodes in range. Messages are encrypted end-to-end. History shows previous received messages with colour-coded type indicators.

05 Map & Node Colours Explained

Node Dot Colours

What each colour means on the map
Teal — BLE + Wi-Fi both active (best connectivity, longest range)
Green — Bluetooth only active (short range, low battery use)
Blue — Wi-Fi Direct only (longer range, higher battery drain)
Grey — All radios off (battery critical or manually disabled)
Large ring — Relay node (higher-capability device, longer range)
Orange ring + glow — Currently selected node (route source)

Edge Line Colours (lines between nodes)

Lines connecting nodes show the active link protocol
Teal dashed — Both nodes have BLE + Wi-Fi active
Blue dashed — Wi-Fi Direct link
Green dashed — Bluetooth link
Grey faint — Link exists but radios are off on one end
Orange solid thick — Active AI-computed rescue route

GPS Status Badge (toolbar)

GPS indicator in the map toolbar
GPS ±Xm (blue) — GPS fix acquired, accuracy shown in metres
GPS… (orange) — Acquiring GPS signal, please wait
No GPS (red) — GPS denied or unavailable on this device
06 Sending an SOS Alert
Emergency use only. Every SOS creates an entry in the rescue team's activity log and triggers the AI routing engine. Do not test during a live operation.

From the Mobile App (Alert Tab)

1
Tap the Alert tab
Second icon in the bottom navigation bar (bell icon with pulsing ring).
2
Select the emergency type
Tap one of the 6 options: War Zone · Flood · Medical · Fire · SOS · Evacuate. The chosen card highlights with its colour. Tap again to deselect.
3
Type a message (strongly recommended)
Tap the message box and type your situation. Include:
  • Your exact location (street, building, landmark)
  • Number of people affected
  • Severity (critical / stable)
  • What you need (ambulance, food, rescue team)
Max 280 characters. Your GPS coordinates are attached automatically.
4
Tap SEND SOS
Button activates (glows red) only after selecting a type. Tap it. The screen shows "SOS Sent — Broadcasting to all reachable mesh nodes". The rescue team's Activity Log updates within 10 seconds.
Offline note: If the backend is unreachable, the UI still shows success. The SOS is queued and will be delivered when a relay node is in range.

From the Command Dashboard (Left Panel)

1
Use the SOS Input Portal in the left panel
Same 3 steps: select type → type message → press SEND SOS. The dashboard version also automatically triggers AI route calculation from the selected node to the nearest relay.
07 AI Route Finding

The AI routing engine finds the optimal multi-hop path from any node to the rescue camp, prioritising nodes with high battery and strong signal.

1
Click a node on the map to select it as the route source
An orange banner appears above the map: "Route source: [node-id] — Send SOS to auto-route from this node". The selected node glows with an orange double-ring on the map.
2
Send an SOS from the SOS Input Portal
The dashboard automatically calls the AI routing engine (POST /api/route) from the selected node to the nearest relay node. Route calculation takes 1–3 seconds.
3
Read the route result strip
A green strip appears below the map showing:
  • Full path: node-A → relay-B → cmd-hq
  • Number of hops
  • Estimated latency in milliseconds
  • Active scenario
An orange line draws the path on the map with an animated white dot travelling along it.
!
If route shows "No route found"
The mesh between the source and destination is broken — no chain of nodes connects them. Solutions:
  • Move a relay node physically closer to fill the gap
  • Trigger HQ Emergency Broadcast to activate all radios
  • Change the disaster scenario to one with a larger radio range
08 HQ Emergency Broadcast
This is a major command action. It activates ALL device radios simultaneously, forces every node to maximum scanning mode, and uses significantly more battery across all devices. Use when you need maximum mesh coverage immediately.
1
Select the correct disaster scenario first
In the Disaster Control Panel (top of left panel), tap the matching scenario button: Flood (blue, 200m Wi-Fi range) · War Zone (red, 50m range) · Earthquake (orange, 400m range). The selected scenario affects the AI routing weights and radio range calculations.
2
Tap TRIGGER HQ EMERGENCY BROADCAST
The large red button at the bottom of the Disaster Control Panel. It shows "BROADCASTING…" with a pulsing radio icon for 1 second, then turns green with "ALL NODES ACTIVATED" for 4 seconds. The map dots all turn teal (BLE + Wi-Fi both on) for 10 seconds.
3
Confirm the Activity Log entry
The bottom Activity Log shows an orange [broadcast] entry: "DISASTER PROTOCOL BROADCASTED — Scenario: EARTHQUAKE — All device radios forced into active BLE scanning mode"
09 Onboarding New Victims (No App Installed)

If a victim has no MeshNet app, use this procedure to get them connected in under 60 seconds.

1
Relay device creates a Wi-Fi hotspot
On the relay phone or laptop, enable the Wi-Fi hotspot. Name it MESHNET-xxxxxx (no password, or a simple shared password written on a card).
2
Victim connects their phone to the hotspot
Victim opens Wi-Fi settings → selects MESHNET-xxxxxx → connects. Takes ~10 seconds. No internet needed — this is a local network between the two devices.
3
Victim opens browser → types relay IP
In any browser (Chrome, Safari, Firefox): go to http://192.168.49.1:4173. The full MeshNet AI app loads in 3–5 seconds directly from the relay's Vite preview server. No internet. No app store. No account.
Shortcut: Show the victim a QR code pre-generated for this URL. Free QR generators work offline. Print it and tape it to the relay device before deployment.
4
Victim installs the PWA (optional but recommended)
Browser shows "Add to Home Screen" / "Install app" prompt. Victim taps it → installed in 10 seconds. Now they have the full app, BLE advertising starts, and they become a mesh node that extends range for others.
5
Victim appears as a dot on the rescue map
The useMeshDiscovery hook fires POST /api/mesh/register with the victim's GPS. Within 10 seconds, their dot appears on the command dashboard map. The rescue team can now track and route to them.
10 Troubleshooting
ProblemCauseFix
Map shows no tiles (grey squares) No internet for OSM tiles Tiles are cached after first load. Press Fit Nodes button. Pre-cache tiles by loading the map while online before deployment.
No nodes on map Backend not running or wrong URL Confirm Terminal 2 (port 4000) is running. Check VITE_API_BASE_URL in backend/config/.env matches the relay's IP.
"Router offline" in top bar Python FastAPI not running Start Terminal 1 (uvicorn api_server:app --port 5050). Route buttons will still work once it's back.
"Stream offline" in top bar SSE connection dropped Refresh the page. SSE reconnects automatically every few seconds.
Map dots in wrong city Seed coordinates are wrong Check database/seeds/nodes.json — coordinates should be Cebu City (~10.31°N, 123.88°E). Re-run the seed script.
SEND SOS button greyed out No emergency type selected Tap one of the 6 type buttons (War Zone, Flood, Medical, Fire, SOS, Evacuate) to activate the send button.
GPS shows "No GPS" (red) Browser location permission denied Click the location icon in the browser address bar → Allow location access. On mobile: Settings → Apps → Browser → Permissions → Location → Allow.
Victim can't connect to hotspot IP Wrong IP or preview server not running Run pnpm preview (not pnpm dev) in Terminal 3 for the preview server on port 4173. Check hotspot IP in network settings — it may be 192.168.43.1 on some Android devices.
Map won't display at all (white/blank) Leaflet container height = 0 Hard refresh the page (Ctrl+Shift+R). If on mobile, make sure you are on the Map tab (not scrolled content). The map requires a non-zero container height.
